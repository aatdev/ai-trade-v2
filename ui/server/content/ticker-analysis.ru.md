# Ticker Analysis — комплексный разбор одного тикера

## Кратко

> Полный end-to-end разбор **одной акции**: новостной фон + фундаментал + технический анализ через живой график TradingView. На выходе — пакет из 4 markdown-файлов, daily/weekly скриншоты и сводный отчёт с конкретными точками входа/выхода (Trigger / Stop / T1–T3) и явной рекомендацией ПОКУПАТЬ / ДЕРЖАТЬ / ПРОДАТЬ.

Скил ничего не дублирует — он **координирует** уже установленные специализированные скилы (market-news-analyst, us-stock-analysis, technical-analyst, chart-analysis) и собирает их результаты в единую структуру. Нужен запущенный **TradingView Desktop** (живой график через MCP на CDP `:9222`).

**Алерты в TradingView по умолчанию НЕ создаются** — только если пользователь явно попросил («создай алерты», «настрой триггеры», «sync alerts», «+alerts»). Без явной просьбы шаг с алертами пропускается целиком.

## Зачем в плане

Это «разбор» — углублённый анализ конкретного кандидата:

- **Шаг 5.2 (лонг)** — на `allow`-режиме автоматика прогоняет top-3 VCP-кандидатов через ticker-analysis: полный разбор подтверждает или снимает кандидата перед постановкой в watchlist.
- **Шаг 6.2 (шорт)** — то же для шорт-ветки в слабом рынке.

После разбора идёт **reconcile**: финальный сигнал (из `signals.md`) сверяется с кандидатом из screener-watchlist — при расхождении (флип направления, обновлённые уровни) запись правится на месте, анализ считается авторитетным.

## Терминология

- **Сигнальный блок** — запись в `trading-data/analysis/signals.md` с уровнями Trigger / Stop / T1 / T2 / T3 приоритетного сценария; единая точка трекинга всех выданных рекомендаций.
- **Приоритетный сценарий** — Long или Short, выбранный по совокупности новостей + фундамента + техники; именно его уровни уходят в `signals.md` (и, опционально, в алерты).
- **reconcile** — сверка сигнала анализа с watchlist-кандидатом скринера; анализ авторитетен, исходные значения скринера сохраняются под `screener_origin`.

## Как работает

**Вход:** тикер (нормализуется в верхний регистр без префикса биржи — `BSX`, не `NYSE:BSX`) и текущая дата сессии. Каталог результатов: `trading-data/analysis/TICKER/YYYY-MM-DD/`.

**Быстрый путь — кэш метрик.** Перед сбором проверяется снапшот `node scripts/read_metrics.js TICKER`: при свежем кэше (≤2 дней) индикаторы, фундаментал и OHLCV берутся оттуда, что снимает часть MCP-вызовов. Скриншоты в кэше не хранятся — снимаются с живого графика всегда.

Три ветки сбора **независимы и идут параллельно**:

1. **Новости → `news.md`.** Методология market-news-analyst, сужена до тикера и окна 10–14 дней. `WebSearch`/`WebFetch`: earnings, корпоративные катализаторы (FDA/SEC/lawsuit/M&A), рейтинги аналитиков, секторные/макро-факторы. Ранжирование по impact-score, 4–6 ключевых событий, итоговый sentiment.
2. **Фундаментал → `fundamental.md`.** Методология us-stock-analysis. Жёсткие метрики — через `mcp__tradingview__fundamentals_get` (мультипликаторы, маржи, баланс, FCF, дивиденды + годовые/квартальные ряды). Web — только то, чего тул не отдаёт: гайденс, сегменты, beat/miss vs консенсус, риски. Сравнение с 2–4 пирами, вердикт.
3. **Техника → `technical.md` + скриншоты.** Методология technical-analyst + chart-analysis через живой график:
   - Weekly: символ → таймфрейм `"W"` → индикаторы (EMA 20/50/200, RSI, MACD, Stochastic, Volume) → `data_get_study_values` → `data_get_ohlcv` (`summary: true`, count 30) → скриншот `TICKER_weekly.png`.
   - Daily: таймфрейм `"D"`, те же индикаторы, OHLCV count 60, скриншот `TICKER_daily.png`.
   - Премаркет — если US-акция и включена расширенная сессия: интрадей `"15"`, gap%, премаркет high/low, относительный объём; таймфрейм возвращается обратно.

**Сведение (после трёх веток):**

4. **`report.md`** — главный файл: новостной фон, фундаментал, техническая фаза + **обе** таблицы сценариев (Long + Short) с конкретными ценами Trigger/Entry/Stop/T1/T2/T3, помеченным приоритетом, главным риском и R/R.
5. **`signals.md`** — обновление журнала: сначала `prune_signals.mjs --ticker TICKER` (удаляет старые блоки этого тикера), затем дозапись свежего блока в строгом формате под `parse_signals.mjs` (заголовок `## DATE — TICKER — 🟢/🟡/🔴 STATUS`, поля `**Trigger для Long/Short:**`, `**Stop:**`, `**T1 / T2 / T3:**`).
6. **(Опц.) Алерты** — только при явной просьбе: `signals-alerts` sync строго по текущему тикеру (`--tickers TICKER`): parse → diff-delete (`--keep-from-plan --message-not-contains "[WL]"`) → create. Watchlist-алерты `[WL]` вечернего синка не трогаются.

**Выход:** 4 markdown + 2 PNG в каталоге тикера, обновлённый `signals.md`, компактный emoji-summary в чат с рекомендацией и планом входа.

## Команда

Плановый headless-вызов (через `claude -p`, Opus 4.8, с vendored TradingView MCP):

```bash
claude -p "Проанализируй тикер <ТИКЕР> (skill ticker-analysis)" \
  --mcp-config .mcp.json --strict-mcp-config \
  --output-format stream-json
```

В чате/Desktop достаточно естественной фразы: «проанализируй BSX», «дай полный разбор NVDA». Алерты — отдельной явной просьбой: «...и создай алерты».
