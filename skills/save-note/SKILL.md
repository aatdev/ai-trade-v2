# /save-note — Сохранение в личную базу знаний

Сохраняет переданный контент в базу знаний MyNotes в структуру `КАТЕГОРИЯ/СУБКАТЕГОРИЯ/`,
автоматически классифицируя материал. Категория и субкатегория подбираются по смыслу контента.

## Где лежит база (настраиваемо)

Базовый каталог определяется так (в порядке приоритета):
1. Переменная окружения `MYNOTES_DIR`, если задана.
2. `MYNOTES_DIR=...` из `.envrc` — сначала в текущем каталоге, затем в корне git-репозитория.
   Поддерживаются обе формы записи: `MYNOTES_DIR=/path` и `export MYNOTES_DIR="/path"`;
   кавычки и ведущая `~` обрабатываются.
3. Фолбэк по умолчанию: `~/Documents/MyNotes`.

Чтение `.envrc` напрямую обязательно: bash-сессии Claude Code не проходят через direnv hook,
поэтому переменная из `.envrc` сама в окружение не попадает. Канонический блок резолва
(использовать его в начале работы, дальше везде только `"$NOTES_DIR"`):

```bash
NOTES_DIR="${MYNOTES_DIR:-}"
if [ -z "$NOTES_DIR" ]; then
  for f in ./.envrc "$(git rev-parse --show-toplevel 2>/dev/null)/.envrc"; do
    [ -f "$f" ] || continue
    v=$(sed -n -E 's/^(export[[:space:]]+)?MYNOTES_DIR=//p' "$f" | tail -1 | sed -E "s/^[\"']//; s/[\"']\$//")
    if [ -n "$v" ]; then NOTES_DIR="${v/#\~/$HOME}"; break; fi
  done
fi
NOTES_DIR="${NOTES_DIR:-$HOME/Documents/MyNotes}"
echo "NOTES_DIR=$NOTES_DIR"
```

Свой путь задаётся вне git: в gitignored `.envrc` проекта, в профиле шелла или в блоке `env`
файла `.claude/settings.local.json`.

**Если каталог недоступен** (например, `.envrc` указывает на несмонтированный сетевой том и
`mkdir -p "$NOTES_DIR"` падает) — не сохраняй молча в другое место: сообщи пользователю, какой
путь не доступен, и спроси, сохранить ли в фолбэк `~/Documents/MyNotes`.

## Что сохранять

Контент берётся из (в порядке приоритета):
1. Аргумент команды / явно указанный пользователем текст или файл.
2. Результат только что выполненной работы в текущем разговоре (последний отчёт, анализ, сводка, код).
3. Если непонятно что именно сохранять — кратко спроси у пользователя.

## Рабочий процесс

### 1. Изучи существующую структуру (ОБЯЗАТЕЛЬНО, первым делом)

Выполни канонический блок резолва `NOTES_DIR` (см. выше), затем:

```bash
mkdir -p "$NOTES_DIR"
find "$NOTES_DIR" -maxdepth 2 -type d | sort
```

Это нужно, чтобы **переиспользовать уже существующие категории/субкатегории**, а не плодить
дубликаты («Финансы» vs «Деньги» vs «finance»). Если по смыслу подходит существующий каталог —
используй его. Новый создавай только если ничего близкого нет.

### 2. Подбери категорию и субкатегорию

- **КАТЕГОРИЯ** — широкая тема верхнего уровня (например: `Здоровье`, `Работа`, `Финансы`,
  `Технологии`, `Обучение`, `Личное`, `Путешествия`, `Дом`).
- **СУБКАТЕГОРИЯ** — уточнение внутри категории (например: `Здоровье/Анализы`,
  `Технологии/Claude-Code`, `Работа/Spitch`, `Финансы/Налоги`).

Правила именования каталогов:
- Кириллица допускается; слова разделяй дефисом, без пробелов (`Claude-Code`, а не `Claude Code`).
- Существительное в именительном падеже, с заглавной буквы.
- Если контент явно одноуровневый и субкатегория не нужна — используй `Общее` как субкатегорию
  (`КАТЕГОРИЯ/Общее`), чтобы структура оставалась двухуровневой `CAT/SUB_CAT`.

Если пользователь дал подсказку категории в аргументе — следуй ей, но всё равно нормализуй
под существующую структуру.

**Исключение — анализы тикеров.** Отчёты по конкретной акции сохраняй в четырёхуровневую
структуру с группировкой по тикеру и дате:

```
Анализ-тикеров/<TICKER>/<YYYY-MM-DD>/<описание>.md
```

Например: `Анализ-тикеров/GOOG/2026-06-05/сводный-анализ.md`. Дату в имени файла
внутри датированного каталога не дублируй. Сохраняй **все** файлы отчёта, не только сводку:
`report.md` → `сводный-анализ.md`, `fundamental.md` → `фундаментал.md`,
`technical.md` → `техника.md`, `news.md` → `новости.md`.

### 3. Сформируй имя файла

- Формат: `YYYY-MM-DD_краткое-описание.md` (дата — сегодняшняя, описание в kebab-case).
- Имя должно отражать суть («2026-06-02_итоги-консилиума.md»), быть коротким и осмысленным.
- Если файл с таким именем уже есть — добавь суффикс `-2`, `-3` и т.д.
- Для анализов тикеров дата уже в пути (`<TICKER>/<YYYY-MM-DD>/`), поэтому имя файла —
  без даты, просто `описание.md` (`сводный-анализ.md`, `техника.md`).

### 4. Сохрани файл

```bash
mkdir -p "$NOTES_DIR/КАТЕГОРИЯ/СУБКАТЕГОРИЯ"
```

Запиши Markdown-файл со шапкой:

```markdown
# <Заголовок заметки>

> Сохранено: YYYY-MM-DD · Категория: КАТЕГОРИЯ / СУБКАТЕГОРИЯ
> Источник: <откуда — разговор, файл, ссылка; если применимо>

<содержимое>
```

### 5. Подтверди результат

Выведи пользователю **полный путь** к сохранённому файлу и одной строкой — выбранную
категорию/субкатегорию и почему. Например:

```
Сохранено: <NOTES_DIR>/Здоровье/Консилиум/2026-06-02_итоги-консилиума.md
Категория: Здоровье / Консилиум (медицинский отчёт по результатам анализа документов)
```

## Принципы

- Не дублируй категории — сначала смотри, что уже есть.
- Структура всегда ровно двухуровневая: `КАТЕГОРИЯ/СУБКАТЕГОРИЯ`
  (исключение — анализы тикеров: `Анализ-тикеров/<TICKER>/<YYYY-MM-DD>/`).
- Не перезаписывай существующие файлы молча — используй суффикс.
- Сохраняй контент как есть (Markdown), не пересказывай и не сокращай без просьбы.
- Резолв базового каталога — всегда через канонический блок (env → `.envrc` → фолбэк);
  не хардкодь `~/Documents/MyNotes` в командах.
