# Ticker Analysis — Сводный анализ одной акции

## Назначение

Этот скил выполняет полный, end-to-end разбор **одного тикера** и формирует пакет файлов, который можно использовать как готовое торговое решение: новости, фундаментал, техника и итоговый отчёт с конкретными точками входа.

**Алерты в TradingView Desktop по умолчанию НЕ создаются.** Если пользователь явно попросил («создай алерты», «настрой триггеры», «поставь алерты», «sync alerts», «+alerts», «и алерты тоже»), скил дополнительно запускает синхронизацию через [[signals-alerts]] в режиме `sync` (diff) на ключевые уровни приоритетного сценария (Trigger / Stop / T1 / T2 / T3): совпадающие алерты остаются нетронутыми, устаревшие удаляются, недостающие создаются.

Скил **координирует** уже существующие специализированные скилы — ничего не дублирует, а вызывает их методологии и собирает результаты в единую структуру.

## Когда триггериться

- «проанализируй TICKER», «сделай аналитику по TICKER», «дай полный разбор TICKER»
- «стоит ли покупать TICKER», «что делать с TICKER», «разбор тикера X»
- «analyze TICKER», «full analysis of TICKER», «give me a report on TICKER»
- любой запрос, где пользователь хочет ОДНОВРЕМЕННО новости + фундаментал + технику по одной бумаге

Не триггериться, если запрос узкий: только новости / только техника / только сравнение нескольких тикеров — в этих случаях работают точечные скилы (`market-news-analyst`, `technical-analyst`, `us-stock-analysis` для compare).

## Структура результата

Всё сохраняется в `reports/analysis/TICKER/YYYY-MM-DD/`, где TICKER — это тикер в верхнем регистре без префикса биржи (BSX, не NYSE:BSX), а дата — текущая дата из контекста сессии.

```
reports/analysis/TICKER/YYYY-MM-DD/
├── news.md           # новостной фон, ранжированный по импакту
├── fundamental.md    # финансы, мультипликаторы, оценка vs peers
├── technical.md      # weekly + daily ТА (тренд, уровни, индикаторы)
├── report.md         # сводный отчёт с точками входа (главный файл)
├── TICKER_daily.png  # скриншот дневного графика с индикаторами
└── TICKER_weekly.png # скриншот недельного графика с индикаторами
```

Эта структура — реальный шаблон, уже используемый в репо (см. `reports/analysis/BSX/2026-05-15/`). Следуй ему.

Дополнительно: финальный сигнал (BUY/HOLD/SELL + entry/stop) сохраняется в общий журнал `reports/analysis/signals.md` — см. раздел «Сохранение сигнала». Перед дозаписью старые блоки по этому же тикеру удаляются (`prune_signals.mjs`), чтобы в журнале оставалась ровно одна актуальная запись на тикер. Историческая копия каждого анализа остаётся в `reports/analysis/TICKER/DATE/report.md`.

**Sync алертов в TradingView (Шаг 6) — опционально**, только если пользователь явно попросил его создать. По умолчанию шаг **пропускается** — отчёт и журнал готовятся, но в TradingView ничего не пишется.

## Алгоритм работы

### Шаг 0. Подготовка

1. Извлеки тикер из запроса. Нормализуй: `BSX`, не `bsx`/`Boston Scientific`/`NYSE:BSX`.
2. Возьми дату из системного контекста (`Today's date is YYYY-MM-DD` в claudeMd). Это `DATE`.
3. Создай каталог: `mkdir -p reports/analysis/TICKER/DATE`.
4. **Определи, просил ли пользователь алерты.** По умолчанию — НЕТ. Признаки явной просьбы (`alerts_requested = true`):
   - в запросе встречаются фразы: «создай алерты», «настрой алерты», «поставь алерты/триггеры», «алерты тоже», «sync alerts», «with alerts», «+ alerts», «+ sync», «и алерты», «+ триггеры»;
   - пользователь явно сослался на скил `signals-alerts` или попросил «синхронизировать триггеры».

   Если ни одного из этих признаков нет — `alerts_requested = false`, и Шаг 6 пропускается целиком. Не догадывайся «он же наверняка хочет» — нет явной просьбы, нет алертов.

   **Быстрый путь — кэш метрик.** Перед сбором данных проверь снапшот: `node scripts/read_metrics.js TICKER` (из корня репо).
   - Exit 0 (снапшот свежий, ≤2 дней) → в JSON уже лежат значения индикаторов (`indicators`: ema20/50/200, sma50/150/200, rsi14, macd, stoch, bb, atr14, returns), фундаментал (`fundamentals`: те же группы, что отдаёт `fundamentals_get`, + `history`) и ценовая сводка (`price`: last_close, year_high/low, pct_from_52w_high, avg_volume_50d). Бери их оттуда — это снимает необходимость навешивать индикаторы на чарт в Шаге 3 и звать `fundamentals_get` в Шаге 2.
   - **Сырые бары (OHLCV)** тоже в кэше: путь в `_cache.ohlcv.path` (`state/metrics/TICKER/ohlcv.json`, бары oldest-first). Если свежо и баров достаточно — читай дневную серию оттуда вместо `data_get_ohlcv`.
   - Exit 3 (нет снапшота или устарел) → тяни вживую через MCP, как описано в Шагах 2–3.
   - Скриншоты в кэше не хранятся — daily/weekly скрины всё равно снимай с живого графика.
5. Создай задачи через TaskCreate (3 параллельные ветки + сводка + журнал + опционально sync алертов):
   - Сбор новостей
   - Фундаментальный анализ
   - Технический анализ (TradingView)
   - Сводный отчёт + точки входа
   - Очистка старых блоков по TICKER из `reports/analysis/signals.md` + дозапись свежего сигнала
   - Sync алертов в TradingView через скил `signals-alerts` для текущего тикера — **только если `alerts_requested = true`**

### Шаг 1. Новостной анализ → `news.md`

Делегируй методологию скилу **market-news-analyst**, но сузь его до конкретного тикера и периода 10–14 дней.

Используй `WebSearch` и `WebFetch`. Параллельно ищи:
- `"TICKER earnings"` за последний квартал
- `"TICKER news last 10 days"` — общие события
- `"TICKER FDA/SEC/lawsuit/recall/M&A"` — корпоративные катализаторы
- `"TICKER analyst ratings upgrade downgrade"` — sell-side движения
- секторные / макро катализаторы, если они применимы (FOMC, тарифы, отраслевые регуляции)

Если новостной поток объёмный (мегакап на отчётах) — запусти подагент `general-purpose` или `Explore` для параллельного сбора через `WebSearch`. Иначе достаточно 3–5 прямых `WebSearch` вызовов.

**Формат `news.md`** (английский — это норма market-news-analyst):

```markdown
# TICKER News Analysis — START_DATE to END_DATE

**Ticker:** Full Company Name (EXCHANGE: TICKER) | **Price:** ~$XX.XX | **Period:** Last N days

## Top News Items (Ranked by Market Impact)

| # | Date | Event | Sentiment | Impact |
|---|------|-------|-----------|--------|
| 1 | ... | ... | Bullish/Bearish/Neutral | Severe/Major/Moderate/Minor |

## Net Sentiment: POSITIVE/NEGATIVE/MIXED (short-term)

[2–4 предложения о том, что сейчас рынок отыгрывает]

## Key Catalysts & Risks (Next 1–4 Weeks)

**Risks (Bearish):**
- ...

**Catalysts (Bullish):**
- ...

**Trade view:** [одна строка — как новости влияют на ближайшее поведение]
```

Ранжируй по impact score методологии market-news-analyst (см. `.claude/skills/market-news-analyst/SKILL.md` §3) — но не переусердствуй с скорингом для одной бумаги, достаточно 4–6 ключевых событий.

### Шаг 2. Фундаментальный анализ → `fundamental.md`

Используй методологию скила **us-stock-analysis** (раздел «Fundamental Analysis»).

**Источник цифр — сначала TradingView MCP, потом web.** Жёсткие числовые фундаментал-метрики тащи через `mcp__tradingview__fundamentals_get` (символ уже на графике после market-news/technical шагов, либо передай `symbol`):
- `fundamentals_get({ symbol, history: true })` даёт за один вызов: мультипликаторы (P/E, P/S, P/B, EV/EBITDA, P/FCF), TTM-выручку/прибыль/EBITDA, маржи, ROE/ROA/ROIC, баланс (долг, кэш, equity, current/quick ratio, D/E), денежный поток (FCF, операционный CF, capex), дивиденды + **годовые/квартальные ряды** выручки, чистой прибыли и diluted EPS (свежий период первым).
- `WebSearch`/`WebFetch` (Yahoo Finance, Seeking Alpha, IR-раздел компании, последний 10-Q/10-K) оставь **только** для того, чего тул не отдаёт: гайденс, сегменты, beat/miss vs консенсус, аналитические рейтинги/таргеты, качественные риски.

Что собрать:
- последний квартал: выручка, EPS, маржа (из `fundamentals_get` — TTM + квартальный ряд), гайденс и beat/miss (web) — сравни с консенсусом и YoY
- сегменты бизнеса (драйверы и слабые места) — web
- баланс: долг, кэш, FCF, D/E (`fundamentals_get`); interest coverage — web при необходимости
- оценка: P/E, EV/EBITDA, P/B, P/FCF (`fundamentals_get`); Fwd P/E, PEG — web; сравнение с 2–4 ключевыми peers (вызови `fundamentals_get` по каждому peer)
- риски (исковая нагрузка, регуляторика, ключевые продукты) — web

**Формат `fundamental.md`** (русский — согласно `.claude/rules/language.md`):

```markdown
# Company Name (TICKER) — Фундаментальный анализ

**Дата:** YYYY-MM-DD
**Текущая цена:** ~$XX.XX
**Капитализация:** ~$XB

## 1. Результаты последнего квартала (отчёт от DATE)

[Таблица с метриками: Выручка / EPS / маржа / FCF — Факт / Конс. / Y/Y]

[1–2 предложения о beat/miss и реакции рынка]

## 2. Сегменты — драйверы

[Разбивка по сегментам с динамикой]

## 3. Маржа, FCF, долг

[Operating margin, FCF, debt levels, ratios]

## 4. Оценка vs пиры

[Таблица: Fwd P/E / EV/EBITDA / Рост выручки / Комментарий]

## 5. Риски

[Bullet-list ключевых рисков]

## Вердикт

[Бычий / медвежий / нейтральный + 2–3 предложения почему]
```

### Шаг 3. Технический анализ → `technical.md` + скриншоты

Используй методологию **technical-analyst** + **chart-analysis** + живой график через **tradingview-mcp**. Это самая ценная часть — потому что у нас есть прямой доступ к TradingView Desktop.

Алгоритм:

1. **Переключи символ:** `mcp__tradingview__chart_set_symbol` с тикером (можно с биржей, например `NYSE:BSX`).
2. **Сначала Weekly:**
   - `mcp__tradingview__chart_set_timeframe` → `"W"`
   - Добавь индикаторы через `chart_manage_indicator` (полные имена!): `"Moving Average Exponential"` ×3 (20/50/200), `"Relative Strength Index"`, `"MACD"`, `"Stochastic"`, `"Volume"`. Если они уже есть — `chart_get_state` это покажет; не дублируй.
   - `data_get_study_values` → текущие значения всех индикаторов
   - `data_get_ohlcv` с `summary: true` + `count: 30` для последних 30 недель
   - `quote_get` → последняя цена
   - `mcp__tradingview__capture_screenshot` с `region: "chart"` → сохрани как `TICKER_weekly.png` (перенеси / переименуй из `screenshots/` в каталог отчёта)
3. **Затем Daily:**
   - `chart_set_timeframe` → `"D"`
   - те же EMA20/50/200 + RSI + MACD + Stochastic + Volume; `data_get_study_values`
   - `data_get_ohlcv` с `summary: true` + `count: 60`
   - `capture_screenshot` → `TICKER_daily.png`
4. **Опционально**, если на графике есть pine-индикаторы (профайлер, supply/demand-зоны) — пройдись по `data_get_pine_lines` / `data_get_pine_labels` / `data_get_pine_boxes` для извлечения ключевых уровней.

Веди работу через скил `tradingview-mcp` для соблюдения правил context management (`summary: true`, `study_filter` где можно, и т.п.).

**Формат `technical.md`** (русский, две секции W и D + сводный диагноз):

```markdown
**Дата:** YYYY-MM-DD
**Цена (last):** $XX.XX
**Контекст:** [1–2 строки о макро-фоне и связи с новостями]

## 1. Недельный таймфрейм (Weekly)

### Тренд
[направление, сила, слом / подтверждение]

### Ключевые уровни
| Тип | Уровень | Природа |
|---|---|---|

### Индикаторы Weekly
- **EMA20W / EMA50W / EMA200W** — позиция цены и наклон
- **RSI(14)W** — текущее значение, перекупленность/перепроданность
```

### Шаг 4. Сводный отчёт → `report.md` (главный файл)

Это самый важный артефакт. Его структура — копия `reports/analysis/BSX/2026-05-15/report.md`.

```markdown
# TICKER — сводный анализ (YYYY-MM-DD)

Цена: ~$XX.XX (контекст: 52w high/low, движение за период).

## 1. Новостной фон — POSITIVE/NEGATIVE/MIXED (краткосрок)

- [4–6 буллетов, конденсированно из news.md]
- Консенсус аналитиков: rating + средний таргет с %апсайдом

- Вердикт одной фразой

- [3–4 буллета сжатой техники]
- Если был предыдущий план — упомяни, как он отработал

### Long-сценарий (приоритет — если ТА это поддерживает)

| T2 | вторая цель |
| T3 | третья цель |

### Short-сценарий (приоритет / альтернатива)
| | |
|---|---|
| Trigger | ... |
| Entry | ... |
| Stop | ... |
| T1 | ... |
| T2 | ... |
| T3 | ... |

- **1–3 месяца:** ...
```

### Шаг 5. Сохранение сигнала → `reports/analysis/signals.md`

После того как `report.md` готов, **обязательно** обнови общий журнал `reports/analysis/signals.md`. Состоит из двух подшагов: **5.1** — удаление старых блоков по этому же тикеру (в журнале остаётся ровно одна актуальная запись на тикер), **5.2** — дозапись свежего сигнала в конец файла.

Этот журнал — единая точка трекинга всех выданных рекомендаций. Историческая копия каждого анализа всегда лежит в `reports/analysis/TICKER/DATE/report.md` — поэтому удаление старых блоков из `signals.md` не теряет данные, а только сводит журнал к актуальному состоянию. Кроме того, если пользователь попросит алерты (Шаг 6 опционально), скил `signals-alerts` читает именно `signals.md` (`parse_signals.mjs`) — поэтому формат должен быть строгий, даже если sync прямо сейчас не запускается. Так пользователь сможет позже сказать «теперь поставь алерты», и они корректно сядут на свежий сигнал.

#### Шаг 5.1. Удаление старых блоков по TICKER (`prune_signals.mjs`)

Перед дозаписью запусти helper, который удаляет из `signals.md` все блоки с заголовком `## YYYY-MM-DD — TICKER — STATUS` для текущего тикера (любые предыдущие даты, любой статус):

```bash
node .claude/skills/signals-alerts/scripts/prune_signals.mjs --ticker TICKER
```

Скрипт меняет файл in-place и выдаёт JSON-отчёт в stdout вида `{"removed": N, "kept": M, "ticker": "TICKER", "source": "...", "dry_run": false}`. Если файла нет (`file_missing: true`) — это норма для пустого журнала, переходи сразу к 5.2.

Принципы:
- Удаляются **только** блоки этого тикера; чужие тикеры скрипт не трогает.
- Совпадение по заголовку (case-sensitive — тикеры всегда uppercase), а не по содержимому, поэтому ручные пометки вида «❌ ЗАКРЫТ» в старых блоках уйдут вместе с ними. Если по бизнес-причине нужно сохранить аннотированную старую запись, скажи об этом пользователю до запуска или пропусти 5.1; по умолчанию журнал чистится.
- Структура файла (преамбула + последовательность блоков, разделённых `---`) сохраняется; в конце файла остаётся `\n---\n`.

Если пользователь явно попросил **сохранить историю** в журнале (например: «не удаляй прошлый сигнал по TICKER», «keep history», «append, не перезаписывай») — пропусти 5.1 и сразу делай 5.2 (чистый append, как было раньше).

#### Шаг 5.2. Дозапись свежего сигнала

После 5.1 в `signals.md` точно нет старых блоков по этому тикеру. Дозаписывай свежий блок в конец файла.

#### Формат блока (строго совместимый с `parse_signals.mjs`)

Каждый новый сигнал — это блок в конец файла, отделённый `---`. Если файла нет — создай его с заголовком `# Trading Signals Journal`.

```markdown
## YYYY-MM-DD — TICKER — 🟢 BUY / 🟡 HOLD / 🔴 SELL (короткое описание сетапа в скобках)

- **Фаза:** [одно предложение про сетап — что сейчас на графике]
- **Trigger для Long:** условие активации (close 1D > $XX.XX на vol > NM) — **обязательно** содержит `$XX.XX` (первое долларовое число — цена триггера)
- **Entry (Long):** $XX.XX–$XX.XX
- **Stop:** $XX.XX
- **T1 / T2 / T3:** $XX.XX / $XX.XX / $XX.XX
- **R/R:** X.X
- **Горизонт:** 1–2 нед (T1) / 1–3 мес (T2–T3)
- **Альтернатива (Short):** close < $XX.XX → entry $XX.XX, stop $XX.XX, T1/T2/T3 $XX/$XX/$XX  ← опционально; парсер её игнорирует (по ключевому слову «Альтернатив»)
- **Главный риск:** [одна строка]
- **Полный отчёт:** [`reports/analysis/TICKER/YYYY-MM-DD/report.md`](./../analysis/TICKER/YYYY-MM-DD/report.md)
```

**Критичные требования к формату** (иначе `parse_signals.mjs` положит сигнал в `skipped`):

1. **Заголовок** — ровно `## YYYY-MM-DD — TICKER — STATUS`, где разделители — em-dash `—` (можно и обычный `-`), TICKER — заглавные буквы/цифры/точка/дефис. Эмодзи статуса задают направление: 🟢 BUY → LONG, 🔴 SELL → SHORT. 🟡 HOLD — direction подхватится из строки Trigger ниже.
2. **Строка Trigger** — должна начинаться ровно с `**Trigger для Long:**` или `**Trigger для Short:**` (направление прямо в названии поля; не пиши `**Trigger:**` без направления — парсер его не подхватит). Первое долларовое число в значении — это цена триггера.
3. **Stop** — поле `**Stop:**`, первое долларовое число в значении — цена стопа. **Не путай со строкой «Альтернатива»** — у неё внутри тоже есть `stop`, но парсер пропускает любую строку со словом «Альтернатив».
4. **T1 / T2 / T3** — одна строка `**T1 / T2 / T3:** $XX.XX / $XX.XX / $XX.XX`. T1 обязателен; T2/T3 могут отсутствовать — план будет на 3–4 алерта вместо 5. Допустимы варианты `**T1 / T2:** ...` или `**T1:** ...`.
5. **Разделитель блоков** — строка `---` на отдельной линии между сигналами.

Цифры в Trigger / Stop / T1 / T2 / T3 бери **ровно** те, что в таблице приоритетного сценария в `report.md`. Если приоритет — Long, заполняй `**Trigger для Long:**` и одноимённый Entry; если Short — `**Trigger для Short:**`. Альтернативный сценарий пиши через `**Альтернатива (Long|Short):**` — он остаётся в журнале как контекст, но алерты на него не создаются.

Если рекомендация — HOLD без чёткого направления (нет приоритетного сетапа): в заголовке статус `🟡 HOLD (сигнала нет)` или `⚪ WAIT (no trade)` — парсер положит сигнал в `skipped` с причиной «нет приоритетного сетапа», и Шаг 6.3 (create) автоматически пропустится. Это **корректное** поведение для HOLD — пиши именно так, не имитируй направление, которого нет.

Этот журнал — единая точка для трекинга всех выданных рекомендаций. Старые блоки по этому же тикеру уже удалены в 5.1 (если пользователь не попросил сохранить историю) — поэтому в файле должна остаться ровно одна актуальная запись на тикер. Другие тикеры не трогай.

### Шаг 6. (Опционально) Обновление алертов через `signals-alerts` (sync) — только по явной просьбе

**Этот шаг выполняется ТОЛЬКО если `alerts_requested = true` (см. Шаг 0.4).**

По умолчанию шаг пропускается полностью: никакие `mcp__tradingview__alert_*` инструменты и `signals-alerts` скрипты не вызываются. В `report.md` блок «🔔 Алерты» **не добавляется** (вместо него см. ниже про «🔔 Алерты не запрошены»). В emoji-summary строка про алерты ставится в виде `🔔 Алерты TradingView: — (не запрошены; для создания скажите «создай алерты»)`.

Если `alerts_requested = true`, запусти sync-режим скила `signals-alerts` строго **по текущему тикеру** (фильтр `--tickers TICKER`). Это:

- удалит только устаревшие «наши» алерты по этому тикеру (с префиксом `TICKER:` в message, которых нет в новом плане);
- создаст недостающие;
- оставит нетронутыми совпадающие по `message` алерты (та же цена, тот же уровень, то же направление) — пользователь не потеряет историю срабатываний и не получит дубликаты.

Алерты других тикеров не затрагиваются.

#### Принципы

- **Только приоритетный сценарий.** `parse_signals.mjs` парсит ровно одно направление (LONG или SHORT) из блока сигнала — определяется по эмодзи статуса в заголовке (🟢 BUY → LONG, 🔴 SELL → SHORT) и/или по строке `**Trigger для Long:**` / `**Trigger для Short:**`. Альтернативный сценарий парсер игнорирует. Конфликтующие алерты в обе стороны не создаём.
- **Источник цифр — `signals.md`** (а не напрямую `report.md`). Журнал — единая точка истины; если цифры в `report.md` и `signals.md` разойдутся, скрипт возьмёт `signals.md`. Поэтому Шаг 5 предшествует Шагу 6 и обязан содержать те же уровни, что и приоритетная таблица в `report.md`.
- **Только один тикер за вызов.** В Шаг 6 передавай `--tickers TICKER` — не запускай sync по всему журналу. Это:
  - быстрее (один сигнал — один проход),
  - безопаснее (чужие алерты по другим тикерам не трогаем),
  - детерминированно (результат скрипта мгновенно соответствует только что записанному сигналу).
- **Никаких параллельных вызовов `signals-alerts`.** Скрипт держит свой CDP-сеанс к TradingView. Дай ему завершиться и прочти stdout.
- **Сам не вызывай `mcp__tradingview__alert_create` / `alert_delete`.** Всё через скрипты `signals-alerts` — иначе работа дублируется и UI-диалоги конфликтуют (см. `feedback_create_alerts.md`).

#### Вызов

Шаги — последовательно, в одной Bash-сессии (`./tmp/signals_plan_TICKER.json` — временный файл по правилам репо):

```bash
# 6.1 — парсинг свежезаписанного сигнала в JSON-план (5 алертов на сценарий)
node .claude/skills/signals-alerts/scripts/parse_signals.mjs --tickers TICKER \
  > ./tmp/signals_plan_TICKER.json

# 6.2 — diff-удаление: снести только устаревшие «наши» алерты по TICKER
node .claude/skills/signals-alerts/scripts/delete_alerts.mjs \
  --keep-from-plan --file ./tmp/signals_plan_TICKER.json

# 6.3 — создание недостающих (дедупликация по message пропустит уже существующие)
node .claude/skills/signals-alerts/scripts/create_alerts.mjs \
  --file ./tmp/signals_plan_TICKER.json
```

Захвати stdout каждого вызова — это JSON-отчёты со счётчиками `created` / `skipped` / `deleted` / `kept` / `errors`. Они нужны и для блока «🔔 Алерты» в `report.md`, и для emoji-summary.

**Сохранение layout.** `create_alerts.mjs` в конце сам сохраняет layout графика в TradingView (кнопка Save / Cmd+S) — поле `layout_save` в его JSON-отчёте; отдельных действий не требуется. Если же алерты ставились вручную через `mcp__tradingview__alert_create` (единичный тикер, вне скриптов) — после создания обязательно сохрани layout: `mcp__tradingview__layout_save` (или `mcp__tradingview__ui_keyboard {key: "S", modifiers: ["meta"]}`), иначе маркер-линии и состояние графика останутся в «unsaved changes».

#### Шаблоны `message` (формируются скриптом — не нужно собирать вручную)

`parse_signals.mjs` сам формирует канонические сообщения. Они должны точно совпадать с этим шаблоном, потому что от них зависит дедупликация:

| Уровень | Long | Short |
|---|---|---|
| Trigger | `TICKER: сигнал на покупку (лонг) — Trigger $XX.XX` | `TICKER: сигнал на продажу (шорт) — Trigger $XX.XX` |
| Stop | `TICKER: закрытие позиции по стопу (лонг) — Stop $XX.XX` | `TICKER: закрытие позиции по стопу (шорт) — Stop $XX.XX` |
| T1 | `TICKER: закрытие позиции по T1 (лонг) — $XX.XX` | `TICKER: закрытие позиции по T1 (шорт) — $XX.XX` |
| T2 | `TICKER: закрытие позиции по T2 (лонг) — $XX.XX` | `TICKER: закрытие позиции по T2 (шорт) — $XX.XX` |
| T3 | `TICKER: закрытие позиции по T3 (лонг) — $XX.XX` | `TICKER: закрытие позиции по T3 (шорт) — $XX.XX` |

`price_condition`: для Long — Trigger/T1/T2/T3 `Crossing Up`, Stop `Crossing Down`. Для Short — зеркально.

**Trigger с объёмным фильтром (multi-condition).** Если в строке `**Trigger для Long/Short:**` в `signals.md` указан объёмный порог («на объёме > 40M», «при vol > 25M», «на volume >= 5.9M», «на V > 30M»), `parse_signals.mjs` распознает абсолютное значение (40M → 40 000 000), направление (`>`/`≥` → `Greater Than`, `<`/`≤` → `Less Than`) и положит их в Trigger-алерт как `volume` + `volume_condition`. `create_alerts.mjs` затем создаст **один** Trigger-алерт с двумя условиями (price + volume через «Add condition» в UI), а не два отдельных. В `message` Trigger автоматически добавляется суффикс ` + vol > 40M` — он входит в ключ дедупликации, и при смене порога объёма алерт корректно пересоздаётся через sync. Если объёма в Trigger-строке нет или он нечисловой («≥ avg») — Trigger остаётся одноусловным по цене.

#### Если Entry — диапазон ($X.XX–$Y.YY)

Отдельный Entry-алерт не нужен: Trigger уже покрывает активацию, а Stop — нижнюю границу инвалидации. План `parse_signals.mjs` всегда 5 алертов (Trigger / Stop / T1 / T2 / T3); если T3 не указан в `signals.md` — будет 4.

#### Дозапись блока «🔔 Алерты» в `report.md`

В самый конец `report.md` (после секции «Файлы») добавь блок подтверждения на основе JSON-отчётов из шагов 6.2 и 6.3:

```markdown
## 🔔 Алерты (sync через signals-alerts, приоритетный Long/Short-сценарий)

**Режим:** sync (diff) — устаревшие удалены, недостающие созданы, совпадающие оставлены.

| Уровень | Цена | Условие | Действие при срабатывании | Статус |
|---|---|---|---|---|
| Trigger | $XX.XX (+ vol > NM, если задан) | Crossing Up/Down (+ Greater Than для vol) | сигнал на покупку (лонг) / сигнал на продажу (шорт) | ✅ создан / ✅ без изм. |
| Stop | $XX.XX | Crossing Down/Up | закрытие позиции по стопу (лонг/шорт) | ✅ создан / ✅ без изм. |
| T1 | $XX.XX | Crossing Up/Down | закрытие позиции по T1 (лонг/шорт) | ✅ создан / ✅ без изм. |
| T2 | $XX.XX | Crossing Up/Down | закрытие позиции по T2 (лонг/шорт) | ✅ создан / ✅ без изм. |
| T3 | $XX.XX | Crossing Up/Down | закрытие позиции по T3 (лонг/шорт) | ✅ создан / ✅ без изм. |

Если Trigger — multi-condition (price + volume), это **один** алерт с двумя условиями (в TradingView UI второе условие добавляется через «Add condition»). В строке Trigger таблицы укажи оба порога одним блоком: например, `$428.50 + vol > 40M`, условие — `Crossing Up + Greater Than (Vol)`.

**Итог:** создано N, удалено M, без изменений K. Алерты привязаны к символу `EXCHANGE:TICKER`, таймфрейм 1D. При срабатывании TradingView Desktop выдаст pop-up + звук, в тексте уведомления — действие из колонки выше.
```

Статус каждой строки бери из JSON:
- `created` (в отчёте `create_alerts.mjs`) → `✅ создан`;
- `skipped` / `kept` (дубликат по message — уже был и совпал) → `✅ без изм.`;
- `errors` → `⚠️ не создан` (после таблицы добавь строку «создать вручную: цена, условие»).

#### Ошибки

- **`tv_health_check` падает (нет CDP)** → пропусти весь Шаг 6. В `report.md` в блоке «🔔 Алерты» напиши «TradingView Desktop недоступен — sync пропущен, см. таблицу приоритетного сценария в Шаге 3 для ручной настройки». В emoji-summary укажи `🔔 алерты: ❌ TradingView недоступен`.
- **`parse_signals.mjs` вернул сигнал в `skipped`** (нет Trigger/Stop/T1 или приоритетный сценарий не определён) → проверь свой Шаг 5: ключевая причина — строка `**Trigger для Long:**` / `**Trigger для Short:**` отсутствует или написана без «для Long/Short» (`**Trigger:**` парсер не подхватит), либо отсутствует одно из обязательных полей `**Stop:**` / `**T1 / T2 / T3:**`. Допиши и перезапусти Шаг 6.
- **`create_alerts.mjs` вернул `errors`** для отдельных уровней → пропусти только их, отметь `⚠️ не создан` в таблице, остальные алерты считай успешными. Скрипт уже сам ретраит один раз.
- **Никогда** не вызывай `mcp__tradingview__alert_delete delete_all=true` для «уборки» — это снесёт ВСЕ алерты пользователя, не только этого тикера. Удаление точечное через `delete_alerts.mjs --keep-from-plan`.

#### Когда НЕ запускать Шаг 6 (sync)

- **По умолчанию — всегда.** Если пользователь не попросил алерты явно (`alerts_requested = false`) — Шаг 6 не выполняется ни в каком виде. Никаких `delete_alerts.mjs`, никаких `create_alerts.mjs`, никаких `mcp__tradingview__alert_*` вызовов. В `report.md` вместо блока «🔔 Алерты» добавь короткую секцию:

  ```markdown
  ## 🔔 Алерты

  Не созданы — пользователь не запросил. Уровни Trigger / Stop / T1 / T2 / T3 приоритетного сценария см. в Шаге 3 выше. Чтобы поставить их в TradingView, скажите «создай алерты по TICKER» — будет вызван `signals-alerts sync` по текущему сигналу из `signals.md`.
  ```

- Даже если `alerts_requested = true`, **не запускай** Шаг 6 если:
  - Итоговая рекомендация в `report.md` — **ДЕРЖАТЬ (HOLD)** без чёткого приоритетного сценария (нет активного сетапа). В Шаге 5 в `signals.md` ставь в заголовок `🟡 HOLD (сигнала нет)` или `⚪ WAIT (no trade)` и не пиши строку `**Trigger для Long/Short:**` — `parse_signals.mjs` тогда положит сигнал в `skipped`. В этом случае:
    - всё же запусти 6.2 (`delete_alerts.mjs --keep-from-plan --file ...`) — это удалит **старые** алерты по этому тикеру, оставшиеся от прошлого сигнала, чтобы пользователь не получил уведомление по уже неактуальным уровням;
    - 6.3 (`create_alerts.mjs`) пропусти — план пустой, создавать нечего;
    - в `report.md` напиши «Алерты не созданы — нет приоритетного сетапа (HOLD); старые алерты по TICKER очищены», в emoji-summary поставь `🔔 алерты: — (HOLD, очищено)`.
  - В `report.md` оба сценария помечены как «не рекомендуется» — аналогично HOLD.

## Правила работы

1. **Краткость важнее полноты** — в `report.md` каждый пункт максимум 1–2 строки; детали — в `news.md`/`fundamental.md`/`technical.md`.
2. **Рекомендация обязательна** — никогда не заканчивай отчёт без чёткого ПОКУПАТЬ/ДЕРЖАТЬ/ПРОДАТЬ. Размытые «возможно стоит присмотреться» не считаются.
3. **Конкретные цифры** — все entry/stop/T1/T2/T3 указаны как реальные цены, а не «около поддержки». То же для уровней индикаторов: «RSI 28», а не «RSI низкий».
4. **`summary: true` для OHLCV** — `data_get_ohlcv` всегда вызывай с `summary: true`, иначе зальёшь контекст сотнями свечей.
5. **Не вызывай `pine_get_source`** — этот тул может вернуть 200KB+ Pine-кода; для одной аналитики он не нужен.
6. **`chart_get_state` — один раз в начале** каждого таймфрейма. Сохрани entity ID индикаторов и используй их в последующих `indicator_set_inputs` / `data_get_study_values`.
7. **Полные имена индикаторов** при добавлении через `chart_manage_indicator`: `"Relative Strength Index"`, `"Moving Average Exponential"`, `"MACD"`, `"Stochastic"`, `"Volume"` — короткие имена («RSI», «EMA») молча не сработают.
8. **Язык** — итоговый чат-summary и `report.md`/`fundamental.md`/`technical.md` строго на русском (правило `.claude/rules/language.md`). `news.md` оставляем на английском (методология `market-news-analyst`).

## Обработка ошибок

- **Тикер не найден в TradingView** → попробуй явный префикс биржи: `NASDAQ:TICKER`, `NYSE:TICKER`, `MOEX:TICKER`, `AMEX:TICKER`. Если не помогло — `symbol_search` для поиска корректного формата.
- **Нет данных по индикаторам** (`data_get_study_values` пустой) → проверь, что индикатор реально добавлен (`chart_get_state`), и что нужный таймфрейм активен. Если индикатор не загрузился — продолжай с OHLCV + скриншотом, отметь ограничение в `technical.md`.
- **TradingView недоступен** (нет CDP-соединения, `tv_health_check` падает) → не блокируй задачу. Выполни шаги 1 (новости) и 2 (фундаментал), пропусти 3. В `report.md` явно укажи: «Технический анализ недоступен — TradingView не подключён», и в emoji-summary поставь рекомендацию ДЕРЖАТЬ, если новости + фундаментал не дают однозначного сигнала.
- **`market-news-analyst` недоступен / `WebSearch` rate-limited** → собери минимум 3–4 новости прямыми `WebFetch` к Yahoo Finance / investor relations компании. В `news.md` пометь «частичные данные».
- **Данных недостаточно для уверенной рекомендации** → дай рекомендацию ДЕРЖАТЬ с явным пояснением, какого слоя данных не хватает, и какое событие/уровень снимет неопределённость.

## Параллелизация

Шаги 1, 2 и 3 **независимы** и могут выполняться параллельно:

- Шаг 1 (новости) → отдельный подагент `general-purpose` если нужно много `WebSearch` за раз; иначе inline
- Шаг 2 (фундаментал) → подагент `general-purpose` для сбора данных с веба, или inline через `WebSearch`/`WebFetch`
- Шаг 3 (техника) → **inline**, потому что использует mcp__tradingview__* (один CDP-канал, нельзя из подагента, который не имеет доступа к этому MCP)

Делай все три ветки в **одном сообщении** (несколько tool calls одновременно). Шаг 4 (`report.md`) делается **после** трёх предыдущих. Шаг 5 (журнал в `signals.md`) — **строго после** Шага 4, потому что цифры приоритетного сценария берутся из готового `report.md`. Внутри Шага 5 — строгий порядок: 5.1 (`prune_signals.mjs --ticker TICKER`) → 5.2 (дозапись свежего блока), иначе `prune` снесёт в том числе только что добавленную запись. Шаг 6 (sync алертов через `signals-alerts`) — **опционально и только при `alerts_requested = true`**, строго после Шага 5.2, потому что `parse_signals.mjs` читает именно `signals.md` и без свежей записи увидит либо старый сигнал, либо ничего. Внутри Шага 6 три bash-команды (parse → diff-delete → create) идут **последовательно**, каждая держит CDP-сеанс к TradingView.

## Чек-лист готовности

Перед тем как сказать пользователю, что отчёт готов, проверь:

- [ ] Все 4 .md файла существуют в `reports/analysis/TICKER/DATE/`
- [ ] Оба скриншота (weekly, daily) лежат там же под правильными именами
- [ ] В `report.md` есть **обе** таблицы сценариев (Long, Short) с конкретными числами entry/stop/T1/T2/T3
- [ ] В `report.md` явно отмечено, какой сетап в приоритете
- [ ] Указан главный риск, который инвалидирует основной сценарий
- [ ] R/R приоритетного сценария посчитан и упомянут
- [ ] Шаг 5.1 — запущен `prune_signals.mjs --ticker TICKER`, старые блоки этого тикера удалены из `signals.md`, JSON-отчёт зафиксирован (если пользователь явно попросил «сохрани историю» — этот пункт можно пропустить и сделать чистый append).
- [ ] Шаг 5.2 — свежий блок дозаписан в `reports/analysis/signals.md` в формате под `parse_signals.mjs`: заголовок `## DATE — TICKER — 🟢/🟡/🔴 STATUS`, поле `**Trigger для Long:**` или `**Trigger для Short:**`, поля `**Stop:**`, `**T1 / T2 / T3:**` (формат строгий — даже если sync сейчас не запускается; иначе при последующем запросе «создай алерты» парсер не подхватит)
- [ ] **Определено**, попросил ли пользователь алерты (`alerts_requested`). По умолчанию `false`.
- [ ] Если `alerts_requested = false` — Шаг 6 пропущен целиком; в `report.md` секция «🔔 Алерты» с пояснением «не запрошены»; в emoji-summary строка `🔔 Алерты TradingView: — (не запрошены)`.
- [ ] Если `alerts_requested = true` — запущен `signals-alerts` sync **только по этому тикеру** (`--tickers TICKER`): parse → diff-delete (`--keep-from-plan`) → create; JSON-отчёты захвачены; блок «🔔 Алерты (sync через signals-alerts...)» в `report.md` с таблицей статусов; для HOLD — выполнен только 6.2 (diff-delete).
- [ ] В чат выдан компактный emoji-summary с рекомендацией ПОКУПАТЬ/ДЕРЖАТЬ/ПРОДАТЬ и корректной строкой про алерты (запрошены / не запрошены).

После проверки отдай пользователю:
1. Компактный emoji-summary (Шаг 6) — это первое, что он увидит
2. Путь к `report.md` (главный файл с деталями)

## Что НЕ делать

- Не делай отчёт без скриншотов — они дают пользователю мгновенный визуальный контекст.
- Не пиши «нужны дополнительные данные» в report.md — если данных не хватает, добери их (доп `WebSearch`, доп `data_get_*`), не оставляй пробелы.
- Не давай размытые точки входа («где-то около поддержки») — всегда конкретные числа.
- Не пропускай stop-loss — каждый сценарий обязан иметь явный инвалидирующий уровень.
- Не используй full-region скриншоты — только `chart`, иначе в png попадает лишний UI и фотография становится бесполезной для архива.
- Не вызывай `pine_get_source` — слишком тяжёлый ответ, для аналитики тикера не нужен.
- Не вызывай `data_get_ohlcv` без `summary: true` — это разрушит контекст.
- Не пропускай дозапись в `reports/analysis/signals.md` — без неё журнал сигналов рассинхронизируется с отчётами, и при последующем запросе «создай алерты» (Шаг 6) `parse_signals.mjs` увидит старый сигнал или ничего, потому что читает именно этот файл.
- Не пропускай Шаг 5.1 (`prune_signals.mjs --ticker TICKER`) перед дозаписью — иначе в `signals.md` накопятся дубликаты по одному тикеру, и `parse_signals.mjs` (Шаг 6) подхватит самый верхний (= самый старый) блок вместо свежего, что создаст алерты по уже неактуальным уровням. Пропускать 5.1 можно ТОЛЬКО если пользователь явно попросил сохранить историю в журнале.
- Не запускай `prune_signals.mjs` без `--ticker` или с пустым тикером — скрипт упадёт с exit code 2, но в более вольной формулировке можно случайно снести чужие блоки; всегда явный `--ticker TICKER` для текущего анализа.
- **Не запускай Шаг 6 без явной просьбы пользователя.** По умолчанию алерты в TradingView НЕ создаются. Признаки явной просьбы перечислены в Шаге 0.4 («создай алерты», «настрой триггеры», «sync alerts» и т.п.). При сомнении — НЕ создавай; пользователь дозапросит, если нужно.
- Не вызывай `mcp__tradingview__alert_create` / `alert_delete` / `alert_list` ни в каком виде, если пользователь не попросил алерты. Это касается и «проверочных» вызовов «посмотреть, что уже стоит».
- Если `alerts_requested = false` — не запускай **ни** `parse_signals.mjs`, **ни** `delete_alerts.mjs`, **ни** `create_alerts.mjs`. Никаких «на всякий случай очистим».
- Не запускай Шаг 6 **до** Шага 5 — порядок строгий: сначала запись в `signals.md`, потом sync алертов.
- Не вызывай `mcp__tradingview__alert_create` / `alert_delete` напрямую в Шаге 6 — это удваивает работу скрипта и ломает UI-диалоги. Всё через `signals-alerts/scripts/*.mjs`.
- Не запускай sync без фильтра `--tickers TICKER` — это пройдётся по всему журналу, удалит/создаст алерты по чужим тикерам и сильно замедлит ответ.
- Не запускай несколько скриптов `signals-alerts` параллельно — каждый держит свой CDP-сеанс, конкурентные вызовы ломают UI-диалоги.
- Не создавай алерты на оба сценария сразу (Long + Short) — `parse_signals.mjs` берёт только приоритетное направление из поля `Direction:` в `signals.md`; не пытайся обойти это, дописывая руками.
- Не вызывай `alert_delete delete_all=true` для уборки — это снесёт ВСЕ алерты пользователя, включая чужие. Точечная очистка — `delete_alerts.mjs --tickers TICKER` или `--keep-from-plan`.
- Не заканчивай ответ без emoji-summary (Шаг 7) — пользователь должен увидеть рекомендацию мгновенно, не открывая `report.md`.
